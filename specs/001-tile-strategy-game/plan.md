# Implementation Plan: Tile-Based Strategy Game

**Branch**: `001-tile-strategy-game` | **Date**: 2026-03-14 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-tile-strategy-game/spec.md`

## Summary

Build a browser-based, single-player tile strategy game (Advance Wars-inspired) playable against an AI opponent. The game features procedurally generated maps with five terrain types, three unit types (Scout, Infantry, Artillery), city/town capture for income, fog of war, and a rule-based + utility-scoring AI. The full stack is TypeScript + PixiJS (renderer) + Vite (build) + Vitest (tests), deployed as static files to GitHub Pages. Game logic is strictly separated from rendering to satisfy the multiplayer-ready architecture requirement.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### I. Simplicity First — PASS (with one justified complexity)

**Justified complexity: PixiJS over vanilla HTML5 Canvas**

The constitution requires: "if a simpler alternative exists, it MUST be used instead." Vanilla HTML5 Canvas is the simpler dependency choice (zero bundle cost). It is rejected because implementing equivalent tilemap rendering (sprite batching, dirty-rect optimization, WebGL acceleration, tilemap draw calls) from scratch is estimated at 500+ lines of infrastructure code with ongoing maintenance for a solo developer. PixiJS eliminates this cost at ~450 KB minified without coupling game logic to the library. The total implementation complexity is lower with PixiJS than without it.

All other choices are the simplest option that satisfies requirements:
- Vitest over Jest (simpler config — reuses vite.config.ts)
- Vite over Webpack/esbuild-standalone (near-zero config for a static game)
- Mulberry32 PRNG over a library (10 lines vs an npm dependency)
- Utility-score AI over Minimax (Minimax is intractable at this grid scale)
- Simplex noise + flood fill over WFC (WFC requires authoring adjacency rules for 5 terrain types)

### II. Test-First Development — PASS (process gate)

All game logic in `src/game/` must have failing tests written before implementation begins (Red-Green-Refactor). The `src/game/` layer contains zero PixiJS imports, so all tests run in Node without a browser or canvas shim. Test files are in `tests/game/`. Coverage must not regress across slices.

### III. Vertical Slice Delivery — PASS

Eight user stories in the spec, each with independent acceptance criteria. The tasks file (tasks.md, to be generated by `/speckit.tasks`) must order implementation so each story is independently testable end-to-end before the next begins. Suggested slice order mirrors spec priority: P1 → P2 → P3 → P4 → P5 → P6 → P7 → P8.

### IV. Single-Player First, Multiplayer-Ready — PASS

Architecture enforces clean separation:
- `src/game/` has zero renderer imports — game state is pure logic, no PixiJS dependency
- `GameState` is fully JSON-serializable (no class instances, no functions in state)
- Actions are the single channel for state mutation (input layer dispatches; renderer only reads)
- Adding networked multiplayer later requires: replacing `src/input/input.ts` with a network message deserializer; moving `src/game/` to a shared package. Renderer is unaffected.
- No hard-coded player count in core logic (`PlayerId` is `'player1' | 'player2'`; game loop is player-agnostic)

### V. Browser-Only Execution — PASS

- PixiJS: runs entirely client-side; no Node.js runtime required
- Vite: produces a static `dist/` directory (HTML + JS + assets) deployable to GitHub Pages
- `simplex-noise`: browser-compatible, ~4 KB
- `mulberry32`: inline function, no dependency
- No WebSocket, no fetch to a backend, no localStorage persistence in this version
- GitHub Pages deployment via a 10-line GitHub Actions workflow

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| PixiJS dependency (~450 KB) | Tilemap rendering with WebGL batching for smooth 60 fps tile draws | Vanilla Canvas requires writing 500+ lines of sprite batching, dirty-rect optimization, and tilemap infrastructure from scratch — higher total complexity for a solo dev |
| `simplex-noise` dependency (~4 KB) | Natural-looking terrain clustering with threshold bucketing | Implementing Perlin/Simplex noise from scratch is ~100 lines of non-trivial math that is error-prone and not worth maintaining; cellular automata produces too many isolated single-tile features on small grids |
